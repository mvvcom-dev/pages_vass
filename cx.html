Клиентский департамент = 2 контура (внедрение/адаптация и развитие/деньги/ценность) + 2 оси (K и S). Мы разделяем ответственность так, чтобы не усложнить управление на вашем масштабе: на каждого клиента — один K, на каждый проект — один S, дальше назначения делаются правилами “по таблице”.

2 контура: внедрение/адаптация и развитие/ценность

Контуры нужны, чтобы не было конфликтов “кто над кем”, и чтобы решения принимались быстро: delivery управляет delivery, коммерция управляет ценностью и деньгами.

Контур 1: внедрение & адаптация (delivery)

Запуск, сроки, план работ, риски и договоренности по объему работ. Интеграции, кастом, неопределенность, стейкхолдеры, дедлайны и репутационный риск. Цель — Time-to-Value (срок до первой ценности).

PM S‑сегмент Риски

Контур 2: развитие & деньги & ценность

Донесение ценности, вовлеченность, удовлетворенность, удержание, пролонгации, апсейлы, контроль рисков оплаты. Цель — NRR/GRR (удержание выручки с ростом / без роста).

Менеджер по успеху клиента (CSM) K‑сегмент Prolongation

Чтобы было просто на вашем масштабе: на каждого клиента — один K, на каждый проект — один S. Дальше назначения делаются правилами “по таблице”.

Две оси

1) Две оси — зачем и как не усложнить

Ось K — “важность/коммерческий класс клиента”

  • какого уровня Менеджер по успеху клиента (CSM) ставим,
  • как часто подключаем директора/СЕО,
  • какой режим удержания/пролонгации.

K = про деньги, потенциал роста, стратегичность и управленческий уровень общения.

Ось S — “сложность/риск проекта (delivery)”

  • какого уровня PM ставим,
  • нужен ли тимлид ПМов/эскалации/стиринг,
  • какой ритм управления рисками.

S = про интеграции, кастом, неопределенность, стейкхолдеров, дедлайны и репутационный риск.

Сегментация клиентов

2) Коммерческая сегментация клиентов K1–K5 (без смешения со “сложностью”)

K — это только про коммерческий класс клиента. Сложность внедрения и риски delivery — это отдельная ось S.

Сегментация проектов

3) Классификация проектов S1–S5 (включаем в систему)

Проект перед назначением получает оценку сложности и приоритетности, и на основе этого — S‑сегмент.

Оценка сложности (факторы)

  • интеграции
  • кастомизация
  • стейкхолдеры
  • неопределенность
  • новизна

Оценка приоритетности (факторы)

  • стратегичность
  • деньги
  • жёсткий дедлайн
  • политическая чувствительность
  • репутационный риск
S Определение Комментарий
S1 Поставка терминала (до 3 единиц, без кастомизации ПО) Минимальный delivery‑риск.
S2 Простой, типовой, низкоприоритетный: поставка терминалов с кастомизацией ПО Есть настройки, но без высокой неопределенности.
S3 Либо сложный, либо приоритетный Требуется более плотное управление.
S4 Сложный и приоритетный одновременно Высокий риск → нужен старший PM.
S5 Высокая неопределенность или стратегическая новизна Усиленный контур управления по умолчанию.
Назначения

4) Правила назначения ролей (самая важная часть)

4.1. PM назначается по S‑сегменту (а не по K)

  • S1 → PM не требуется (ведет служба HelpDesk)
  • S2 → PM Junior / Middle (по загрузке)
  • S3 → PM Middle+
  • S4 → PM Senior
  • S5 → PM Senior + обязательное участие тимлида ПМов (в контуре управления)

4.2. Менеджер по успеху клиента (CSM) назначается по K‑сегменту

(роль объединена — ок; финансовую рутину снимаем)

  • K1 → Менеджер по успеху клиента (CSM) Middle (портфельный режим)
  • K2 → Менеджер по успеху клиента (CSM) Middle
  • K3 → Менеджер по успеху клиента (CSM) Senior
  • K4 → Выделенный Менеджер по успеху клиента (CSM) Senior
  • K5 → Выделенный Менеджер по успеху клиента (CSM) Senior + директор CX как executive‑сопровождение + СЕО по триггерам/квартально

4.3. Участие руководства: правило “верхняя граница риска”

  • Тимлид ПМов подключается, если S4/S5 (а также при срывах/эскалациях)
  • Директор CX подключается, если K4/K5 (и top‑K3), а также если высокий риск оттока/конфликт/штрафы
  • СЕО: K5 — планово раз в квартал по ключевым, плюс по триггерам (см. раздел 8)
Роли

5) Роли и границы ответственности (без конфликтов “кто над кем”)

PM (проектный менеджер)

Отвечает за delivery‑контур: план, сроки, риски, договоренности, объем работ (рамки проекта), управление стейкхолдерами по внедрению, качество запуска.

Не отвечает за: апсейлы (допродажи), дебиторку, счета/акты, изменение коммерческих условий.

Метрики PM: Time‑to‑Value · On‑time Delivery · Implementation CSAT/NPS · Scope Creep %

Менеджер по успеху клиента (CSM) (ценность + коммерческий контур)

Отвечает за ценность и деньги: донесение ценности, вовлеченность, удовлетворенность, пролонгации, апсейлы, контроль рисков оплаты (как владелец результата).

Ограничения: не обещает сроки без PM; не управляет delivery; не вмешивается в разработку.

Метрики Менеджера по успеху клиента (CSM): NRR · GRR · Renewal Rate · Expansion · Product/Usage Health · NPS

Финансовый менеджер (исполнение финансовой рутины)

Исполняет операционку: счета, акты, сверки, реестр оплат, контроль дебиторки как процесс (вместе с Менеджером по успеху клиента (CSM) по эскалациям). Важно: Менеджер по успеху клиента (CSM) — владелец результата “клиент оплатил и продлил”, финменеджер — исполнитель документооборота.

Helpdesk (поддержка)

логистику/поставку ведёт партнёр; выезда/монтажа нет; Helpdesk контролирует подключение ПО, доступы, типовые инциденты (L1); сложные интеграции/ошибки — на L2 (инженер/разработка) по регламенту.

Ритмы

6) Ритмы встреч и управление по стадиям (упрощённо и практично)

Стадии клиента в системе: 1) Внедрение · 2) Стабилизация (30–60 дней) · 3) Регулярная эксплуатация и рост · 4) Окно пролонгации (за 90–120 дней до конца).

Стадия Ритм
Внедрение Еженедельно статус (PM ведёт), Менеджер по успеху клиента (CSM) присутствует минимум раз в 2 недели
Стабилизация (первые 30–60 дней) Раз в 2 недели (PM + Менеджер по успеху клиента (CSM)), затем передача в “эксплуатацию”
Регулярная эксплуатация и рост K1–K2: раз в месяц · K3: раз в месяц · K4–K5: раз в 2–4 недели (в зависимости от риска)
Окно пролонгации (за 90–120 дней) QBR для K3–K5 раз в квартал (директор CX как executive + Менеджер по успеху клиента (CSM)), по K5 при необходимости — СЕО

7) Как не путать “клиент K” и “проект S” в ежедневной работе

  • “Кто ведёт сроки и работы?” → смотри S → PM и тимлид ПМов при S4/S5
  • “Кто ведёт деньги/ценность/пролонгацию?” → смотри K → Менеджер по успеху клиента (CSM) и директор CX при K4/K5
  • “Кто подписывает/выставляет/сверяет?” → финменеджер
Эскалации

8) Триггеры эскалаций (когда обязательно подключать руководителей)

  • риск срыва срока > X дней (зафиксировать порог, напр. 10 рабочих дней)
  • конфликт по объёму работ / клиент требует “бесплатно”
  • рост жалоб / резкое падение NPS
  • риск не‑пролонгации в ближайшие 120 дней
  • штрафы/политическая чувствительность/репутационный риск
  • проект S5 (по умолчанию)
Система

9) Мини‑ТЗ для вашей информационной системы (обязательно добавить поля)

Карточка клиента

  • K‑сегмент (K1–K5) — выпадающий список
  • Потенциал роста (да/нет + ожидаемый чек) — обязательное поле для K1
  • Дата окончания договора (для окна пролонгации)
  • Ответственный Менеджер по успеху клиента (CSM)
  • Ответственный директор (executive sponsor) (если K4/K5)
  • Здоровье клиента (зелёный/жёлтый/красный) + причина

Карточка проекта

  • S‑сегмент (S1–S5) — выпадающий список
  • Оценка сложности (5 факторов, шкала 1–5)
  • Оценка приоритетности (5 факторов, шкала 1–5)
  • Стадия (внедрение/стабилизация/эксплуатация/пролонгация)
  • Ответственный PM
  • Тимлид ПМов в контуре (да/нет) автофлаг при S5
  • Риски (реестр) + дата следующего ревью

Финансовый контур

  • Счёт/акт/оплата — ведёт финменеджер
  • Дебиторка (сумма, срок, статус)
  • Флаг “требуется эскалация Менеджера по успеху клиента (CSM)” (если просрочка > N дней)

Терминал (карточка “сделки/заказа терминала”)

  • Тип заказа: Терминал только (S0) / Терминал + пакет (S1–S5)
  • S‑сегмент: S1 / S2 / S3 / S4 / S5
  • PM required: да/нет (авто: S0 = нет)
  • Ответственный Helpdesk
  • K‑сегмент клиента (K1–K5) (подтягивается из карточки клиента)
  • Потенциал апгрейда (да/нет + ожидаемый чек)
  • Чек‑лист подключения (статусы: подготовка → подключено → тест → приемка)

Важно: если K и S не заполнены в карточках — правила не работают. Поэтому поля должны быть обязательными.

Терминал без пакета

10) Поставка терминала самообслуживания без пакета услуг

Helpdesk (владелец операционного исполнения)

  • контроль готовности терминала к подключению (чек‑лист)
  • подключение/активация ПО
  • настройки по шаблону (типовые профили)
  • тест работоспособности (приемочный чек‑лист)
  • передача в поддержку (L1) после приемки

Правило назначения

Заказ чисто на терминал самообслуживания без пакета услуг = S0. Проектного менеджера не требуется. Сразу подключается Helpdesk, который занимается подключением терминала. При этом Менеджер по успеху клиента (CSM) может назначаться разного уровня в зависимости от коммерческой ценности клиента по оси K.

S0 → PM нет K → уровень CSM Helpdesk → владелец исполнения
